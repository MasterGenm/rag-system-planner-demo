## 1. 修订动机（来自 v0.1.0 的两条诚实记录）

### 来源 A：CASE-0003 (Lettria GraphRAG)

> "`must_avoid: graph_rag` 应从硬性避免变成语境化延后。当场景同时具备高风险行业、复杂半结构化文档、强 lineage/auditability 要求、vector-only baseline 明确不足、并且团队能验证 graph extraction 与双写一致性时，`must_avoid: graph_rag` 应从硬性避免变成语境化延后。"

→ 5 项触发条件（同时满足才豁免）：

1. 高风险或受监管行业（金融、法律、医疗、航空航天等）
2. 复杂半结构化文档（含表格、图表、专业术语）
3. 强 lineage / auditability 要求
4. vector-only baseline 已被实测验证不足（含具体数字或可观察指标）
5. 团队具备 graph extraction、双写一致性、payload 索引等工程能力的可验证证据

### 来源 B：CASE-0001 (Court Logic)

> "`must_avoid` 表述应允许在固定 hard-case set 上做 bounded experiment。"

→ 即使豁免不成立，也允许在**固定 hard-case set + 受控条件**下做 bounded experiment 来收集证据，不是直接否决。

### 不在本次修订范围内的发现

- CASE-0002 揭示的 "Skill 单一 `failure_family` 输出需要配套记录次要 failure signals" 留待 v0.2，**本次不处理**。理由：这涉及 Skill 输出 schema 的扩展，不是 must_avoid 条款的细化，影响面更大。

## 3. 变更细节（按文件）

### 3.1 `skills/rag-system-planner/SKILL.md`

**只允许在现有"克制升级"或"决策风格"相关段落附近增加新内容，不删现有内容**。

新增内容必须包含两个子段：

#### 子段 A：语境化升级豁免（5 项触发条件）

中文表述，结构示例（**示例供你参考措辞，最终措辞由你写、与现有文档风格一致**）：

> 默认情况下，向 GraphRAG、multi-agent rewrite 等高复杂度架构的升级应被推迟。
>
> 例外条款：仅当下列 5 项条件**全部**被实测证据证实时，可将"硬性避免"放宽为"语境化延后"，并允许在固定 hard-case set 上做 bounded experiment：
>
> 1. 场景属于高风险或受监管行业（金融、法律、医疗、航空航天等）。
> 2. 语料包含表格、图表、专业术语等复杂半结构化结构。
> 3. 业务输出对 lineage 与 auditability 有明确硬性要求。
> 4. vector-only baseline 已被实测验证不足，并附具体数字或可观察的失败模式。
> 5. 团队具备 graph extraction、双写一致性、payload 索引等工程能力的可验证证据。
>
> 即使豁免成立，**仍不直接生产化**升级方案；豁免的含义是"允许进入受控对照实验阶段"。

#### 子段 B：bounded experiment 的边界

> 当不满足上述 5 项中的任意一项时，仍可对升级方案进行 bounded experiment，但必须满足：
>
> - 在固定 hard-case set 上对照
> - 与 baseline 同时跑，不替换
> - 实验产出明确的 win/loss/inconclusive 判定
>
> bounded experiment 不是升级承诺，是证据收集。

#### 反目标

- 不要把豁免段写成"鼓励升级"的语气
- 不要削弱原有的"默认克制"立场
- 不要在子段 A 加第 6、第 7 项条件——5 项就是 5 项

## 8. 终止条件

如果遇到以下任一情况，**停止动手**，回到发起人确认：

1. 修订 SKILL.md 时发现既有"克制升级"段的措辞与本规格 §3.1 描述不兼容（例如已有段落本身就有"豁免"概念但定义不同）。
2. 修订 score_run.py 时发现现有逻辑无法在不破坏 16 个 self-test 的前提下加入豁免分支。
3. BCH-0017 的设计与 §3.3 的字段约束（如 schema 兼容性）冲突。
4. 任何 case 文件的"v0.1.1 吸收注记"加入后导致 freshness 检查报告 stale。

不要"自行决定继续"。
